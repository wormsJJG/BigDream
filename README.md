BigDream 개발 문화 및 협업 가이드
=======================

더 견고한 코드, 더 안전한 보안을 위한 우리의 약속

1\. Git 브랜치 관리 전략 (Git-flow Lite)
---------------------------------

빅드림의 모든 소스코드는 엄격한 형상 관리를 통해 보호됩니다. 우리는 **Git-flow**를 기반으로 하되, 실무에 맞게 최적화한 전략을 사용합니다.

main : 언제든 배포 가능한 **운영(Production)** 상태. (직접 Push 금지)

develop : 다음 버전을 위한 **개발 통합** 브랜치. 모든 기능이 여기서 합쳐집니다.

feature/\* : 새로운 기능 개발 시 사용. (예: `feature/login-auth`, `feature/adb-scan`)

hotfix/\* : 운영 중 발생한 **긴급 버그 수정**용 브랜치.

⚠️ 절대 원칙 개발자는 `main`이나 `develop` 브랜치에 직접 코드를 Push할 수 없습니다.  
반드시 본인의 작업 브랜치(feature)에서 작업 후 **Pull Request(PR)**를 생성해야 합니다.

2\. Pull Request(PR) & 코드 리뷰
----------------------------

코드 리뷰는 감시가 아닌 **'지식 공유'**와 **'품질 향상'**의 과정입니다. 빅드림의 모든 코드는 최소 1명 이상의 동료 검토를 거쳐야 Merge 됩니다.

### 🚀 PR 프로세스

1.  작업 브랜치에서 기능 개발 완료 및 테스트 수행
2.  Github(또는 GitLab)에서 `develop` 브랜치를 대상으로 PR 생성
3.  **PR 템플릿 작성:** 작업 목적, 변경 내용, 테스트 결과, 특이사항 기재
4.  동료 개발자를 Reviewer로 지정
5.  리뷰어의 **Approve(승인)** 획득 후 Merge 진행

### 🛡️ 리뷰 체크리스트 (보안 기업 특화)

*   **Security:** 하드코딩된 비밀번호/API Key는 없는가? (개인정보 노출 위험 체크)
*   **Logic:** 예외 처리(Try-Catch)는 꼼꼼한가? (앱 크래시 방지)
*   **Readability:** 변수명/함수명은 직관적인가? 주석은 충분한가?
*   **Performance:** 불필요한 반복문이나 메모리 누수 가능성은 없는가?

3\. 업무 능력 향상 및 성장 방향
--------------------

단순히 기능만 구현하는 '코더(Coder)'가 아닌, 문제를 해결하는 '엔지니어(Engineer)'로 성장하기 위한 빅드림의 방향성입니다.

### 📚 문서화의 생활화

"기록되지 않은 코드는 레거시(Legacy)가 된다."  
\- 개발 로직, API 명세서, 트러블슈팅 과정을 Wiki(Notion)에 상세히 기록합니다.

### 🧩 클린 코드 지향

"나만 알아보는 코드는 쓰레기다."  
\- 변수명 하나를 짓더라도 의미를 담고, 함수는 하나의 기능만 수행하도록 쪼개는 습관을 기릅니다.

### 🗣️ 주간 기술 공유회 (Tech Talk)

매주 1회, 자신이 해결한 버그나 새로 배운 기술(스파이앱 패턴 등)을 팀원들과 짧게 공유하는 시간을 갖습니다.

### 🤖 자동화에 대한 고민

반복되는 빌드, 배포, 테스트 과정은 스크립트(CI/CD)로 자동화하여, 창의적인 업무에 집중할 시간을 확보합니다.
